# Feature Specification: 翻译模块重构 — 逐条翻译与实时双语展示

**Feature Branch**: `003-sequential-realtime-translate`  
**Created**: 2026-03-03  
**Status**: Draft  
**Input**: User description: "完全重构整个翻译模块，无需进行并发翻译，只逐条翻译，但是需要实时展示，与原文匹配展示。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 逐条翻译并实时展示双语字幕 (Priority: P1)

用户打开视频页面后触发翻译，系统按字幕顺序逐条翻译：每翻译完一条，立即在视频画面上以"中文译文 + 英文原文"的双语格式展示该条字幕，用户无需等待全部翻译完成即可看到已翻译的内容。

**Why this priority**: 这是核心价值路径——用户最关心的是"翻译一条就看到一条"，而非等全部翻译完毕才出现字幕。

**Independent Test**: 在任意英文字幕视频中触发翻译，视频播放到第一条字幕时间点时，该条已翻译完成并以双语格式显示在画面上。

**Acceptance Scenarios**:

1. **Given** 页面存在英文字幕且翻译已触发，**When** 第一条字幕翻译完成，**Then** 用户立即在视频画面上看到"中文 + 英文"双行字幕。
2. **Given** 翻译正在逐条进行中，**When** 视频播放到某一条尚未翻译完的字幕，**Then** 仅展示英文原文；待翻译完成后自动补上中文行。
3. **Given** 翻译正在逐条进行中，**When** 用户查看控制面板，**Then** 可以看到当前已完成/总数的翻译进度。

---

### User Story 2 - 翻译进度可视化与状态反馈 (Priority: P1)

翻译过程中，控制面板和弹窗需实时展示翻译进度（已完成条数/总条数），用户能清楚地知道翻译正在进行、进度如何，以及是否完成。

**Why this priority**: 在逐条翻译模式下，单次翻译耗时可能较长，用户需要明确的进度反馈以免误判为"卡住"。

**Independent Test**: 触发翻译后，控制面板持续更新"已翻译 N/M 条"信息，翻译完成后显示"翻译完成"。

**Acceptance Scenarios**:

1. **Given** 翻译开始后，**When** 每完成一条字幕翻译，**Then** 控制面板进度更新为"已翻译 N/M 条"，弹窗状态同步更新。
2. **Given** 全部字幕翻译完成，**When** 用户查看状态，**Then** 显示"翻译完成"并自动保存到缓存。

---

### User Story 3 - 翻译失败时逐条降级与错误提示 (Priority: P2)

如果某一条字幕翻译失败（网络中断、API 限流等），系统应跳过该条并继续翻译后续字幕，同时在该条字幕位置标注"翻译失败"，用户可稍后手动重试失败的条目。

**Why this priority**: 保证单条失败不阻塞整体流程，提升用户体验的韧性。

**Independent Test**: 模拟某条翻译请求失败，系统继续翻译后续字幕，失败条目显示提示文案。

**Acceptance Scenarios**:

1. **Given** 翻译过程中某一条请求失败，**When** 重试耗尽，**Then** 该条标记为失败并跳过，继续翻译后续字幕。
2. **Given** 存在翻译失败的条目，**When** 用户重新触发翻译，**Then** 仅重试失败的条目，已成功的不重复翻译。

---

### User Story 4 - 缓存与翻译结果持久化 (Priority: P2)

翻译完成后的结果自动保存到本地缓存；下次打开同一视频时，直接从缓存加载已翻译的字幕，无需重复翻译。

**Why this priority**: 避免每次打开同一视频都重新逐条翻译，节省时间与 API 调用。

**Independent Test**: 翻译完成后关闭页面再重新打开，字幕立即以双语形式显示（缓存命中），无需等待翻译。

**Acceptance Scenarios**:

1. **Given** 翻译已完成并缓存，**When** 用户再次打开同一视频，**Then** 直接加载缓存，无翻译请求。
2. **Given** 字幕内容发生变化（哈希不匹配），**When** 用户打开视频，**Then** 清除旧缓存并重新翻译。

### Edge Cases

- 字幕总数为 0（页面无可用字幕）：应直接拒绝翻译并提示。
- 翻译过程中用户切换视频或刷新页面：应中断当前翻译，不保存不完整结果到缓存。
- API 连续限流导致大量条目失败：进度面板应明确告知"大部分翻译失败"并建议稍后重试。
- 字幕文本为空或仅包含符号/标点：跳过翻译，保留原文。
- 用户在翻译过程中切换翻译服务（Google → OpenAI）：应中断当前翻译并以新服务重新开始。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 按字幕序号逐条发送翻译请求（一次只翻译一条），不得并发或批量发送。
- **FR-002**: 系统 MUST 在每条字幕翻译完成后立即将译文写入对应字幕对象，并触发实时渲染更新。
- **FR-003**: 字幕展示 MUST 以双语格式（中文译文行 + 英文原文行）呈现；若该条尚未翻译完成，仅展示英文原文。
- **FR-004**: 翻译进度 MUST 在控制面板和弹窗中实时更新，格式为"已翻译 N/M 条"。
- **FR-005**: 单条翻译失败时，系统 MUST 在重试耗尽后跳过该条并继续后续翻译，不得阻塞整个流程。
- **FR-006**: 失败的条目 MUST 被记录，用户重新触发翻译时仅重试失败条目。
- **FR-007**: 翻译完成后 MUST 自动将结果保存到本地缓存。
- **FR-008**: 系统 MUST 在翻译过程中禁止并发请求——同一时间只有一条字幕在翻译。
- **FR-009**: 翻译被中断（页面关闭、用户切换视频）时，MUST 不保存不完整翻译到缓存。
- **FR-010**: 空文本或纯符号字幕 MUST 跳过翻译，直接保留原文。

### Key Entities

- **SubtitleItem（字幕条目）**: 单条字幕，包含序号、时间范围、英文原文、中文译文、翻译状态（pending/translating/done/failed）。
- **TranslationSession（翻译会话）**: 一次完整的翻译流程，包含总条数、已完成条数、失败条目列表、是否中断。
- **TranslationResult（翻译结果）**: 缓存结构，包含视频标识、字幕哈希、所有字幕条目的翻译结果、保存时间。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 首条字幕翻译结果在触发翻译后 3 秒内可见（网络正常情况下）。
- **SC-002**: 翻译过程中，每条译文在完成后 200ms 内渲染到画面上。
- **SC-003**: 进度面板的数字与实际已完成条数一致，无延迟超过 500ms。
- **SC-004**: 单条翻译失败不影响后续字幕翻译，整体流程可继续。
- **SC-005**: 缓存命中时，字幕在页面加载后 1 秒内以双语形式完整展示。
- **SC-006**: 重试翻译时，已成功条目不重复请求，仅处理失败条目。

## Assumptions

- 英文字幕是唯一允许的翻译源语言（由上游"英文字幕自动识别"特性保证）。
- 翻译服务（Google 或 OpenAI）的单条请求延迟在合理范围内（通常 < 2 秒/条）。
- 用户可通过现有控制面板和弹窗交互，不引入新的 UI 入口。
- 缓存机制复用现有的 `chrome.storage.local` + 哈希校验方案。
- 本重构不改变翻译服务的选择逻辑（Google/OpenAI 切换由现有配置决定）。
